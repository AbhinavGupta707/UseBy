# Checkpoint 6 Live Smoke

This smoke plan verifies DemandPool and merchant portal behavior after the CP6 migration, backend lanes, UI lanes, and proof wiring are integrated. CP6 is payment-deferred: commitments are unpaid demo intent until payment infrastructure is explicitly reintroduced.

## Local

Run from `useby-app`:

```bash
npm run lint
npm run typecheck
npm run test
npm run build
```

Run from the repository root:

```bash
git diff --check
```

Expected result: checks pass, `/merchant` and `/proof` render, and missing CP6 routes appear as unavailable instead of demo-filled final output.

## Migration

Before applying to production, confirm the CP6 migration adds the runtime-owned output tables and columns expected by proof:

- `demand_pools.awarded_bid_id`, status, committed quantity, committed households.
- `demand_pool_commitments` active/fulfilled commitment state.
- `merchant_bids` price, quantity, pickup window, score, and status.
- `pool_orders` generated from active commitments after award.
- `pickup_tasks` or equivalent pickup/order task rows for ready and collected state.
- `job_runs` rows for `close-demand-pools`.
- `audit_events` rows for pool, bid, award, order, and pickup transitions.

Expected result: seed may create input pools, commitments, and submitted bids, but it must not seed awards, pool orders, pickup tasks, winners, fulfilled commitments, or job output.

## Production

Use the public production alias:

```bash
curl -i https://useby-app.vercel.app/api/system/db-proof
curl -i https://useby-app.vercel.app/api/system/state
curl -i https://useby-app.vercel.app/api/demand-pools
curl -i https://useby-app.vercel.app/api/merchant/demand-pools
curl -i https://useby-app.vercel.app/api/jobs/close-demand-pools
```

Expected result: configured endpoints return JSON. If a route is not deployed yet, it returns an honest unavailable/404 state and the proof UI reflects that.

## API Mutation

From current Aurora rows:

1. Create or choose a gathering pool below threshold.
2. Join with a household that does not already have an active commitment in that pool.
3. Verify committed quantity and committed household count recompute from `demand_pool_commitments`.
4. Submit at least two merchant bids for the same eligible pool.
5. Change price, capacity, pickup window, or substitution terms and verify award scoring can choose a different winner.
6. Run `POST /api/jobs/close-demand-pools`.
7. Verify exactly one winning bid, rejected non-winners, `demand_pools.awarded_bid_id`, generated `pool_orders`, generated `pickup_tasks`, a `job_runs` row, and CP6 `audit_events`.
8. Mark one pickup ready with `POST /api/merchant/pickups/:orderId/ready`.
9. Mark that pickup collected with `POST /api/merchant/pickups/:orderId/collected`.

Expected result: all final states come from live rows generated by these actions, not from seed data.

Expected result: `/merchant` shows active pools, aggregate demand, bid submission/status, awarded pickups, ready/collected controls, route availability, and no-payment copy. No surface exposes exact household coordinates, unit labels, direct email, or phone details.

## Aurora

Inspect sanitized evidence only. Secret ARNs and resource IDs are acceptable; plaintext passwords, tokens, and API keys are not.

Expected evidence:

- `/api/system/state` reports active pools, live commitments, merchant bids, awarded pools, pool orders, pickup tasks, close-pool job runs, and CP6 audit events.
- `/api/system/db-proof` reports Aurora PostgreSQL and extension status.
- Latest audit events include pool commitment, merchant bid, award/order creation, ready pickup, and collected pickup actions after the smoke.
- Payment state is absent or explicitly deferred; API output and browser copy describe commitments as unpaid demo intent.
