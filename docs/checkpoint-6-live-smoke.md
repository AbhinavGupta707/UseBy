## Migration

Before applying to production, confirm the CP6 migration adds the runtime-owned output tables and columns expected by proof:

- `demand_pools.awarded_bid_id`, status, committed quantity, committed households.
- `demand_pool_commitments` active/fulfilled commitment state.
- `merchant_bids` price, quantity, pickup window, score, and status.
- `pool_orders` generated from active commitments after award.
- `pickup_tasks` or equivalent pickup/order task rows for ready and collected state.

## API Mutation

From current Aurora rows:

1. Create or choose a gathering pool below threshold.
2. Join with a household that does not already have an active commitment in that pool.
3. Verify committed quantity and committed household count recompute from `demand_pool_commitments`.
4. Submit at least two merchant bids for the same eligible pool.
5. Change price, capacity, pickup window, or substitution terms and verify award scoring can choose a different winner.
6. Run `POST /api/jobs/close-demand-pools`.
7. Verify exactly one winning bid, rejected non-winners, `demand_pools.awarded_bid_id`, generated `pool_orders`, generated `pickup_tasks`, a `job_runs` row, and CP6 `audit_events`.
8. Mark one pickup ready with `POST /api/merchant/pickups/:orderId/ready`.
9. Mark that pickup collected with `POST /api/merchant/pickups/:orderId/collected`.

Expected result: all final states come from live rows generated by these actions, not from seed data.
