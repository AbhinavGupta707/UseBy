# Checkpoint 7 Live Smoke

Checkpoint 7 proves merchant surplus drops, household reservations, merchant status controls, expiry jobs, and coarse heatmap evidence against live Aurora rows. Seed data may create merchant drop input rows only. Reservations, sold-out or closed states, heatmap cells, expiry job runs, and CP7 audit evidence must be generated from current user, merchant, or job actions.

## Local Readiness

From `useby-app` after all CP7 lanes are merged:

```bash
npm run lint
npm run typecheck
npm run test
npm run build
```

From the repository root:

```bash
git diff --check
```

Expected result: local tests pass, proof and merchant surfaces show unavailable states when CP7 routes or Aurora env are absent, and no plaintext secrets are printed.

## Migration Readiness

Before production smoke, apply the CP7 migration owned by Lane 7A/7B and verify the live schema includes:

- `store_drops` with merchant ownership, status, quantity, pickup-window, safety-note, price-display/demo-intent, and demo-scope metadata.
- `store_drop_reservations` with drop, household, status, quantity, idempotency, cancellation/release, and audit linkage.
- Any Lane 7B heatmap storage, if used, exposed to `/api/system/state` as coarse aggregate cells.
- `job_runs` rows for `expire-store-drops` and `audit_events` rows for drop and reservation mutations.

Expected result: migrations do not add Stripe, payment ledger, deposit, authorization, captured-charge, or paid commitment state.

## Mutation Smoke

Use current live IDs from list responses and a fresh idempotency key for each mutation:

1. Create or update a merchant surplus drop through `/api/merchant/store-drops`.
2. Publish the drop through `/api/merchant/store-drops/:dropId/publish`.
3. Reserve a positive quantity through `/api/store-drops/:dropId/reserve`.
4. Repeat the same reserve request with the same idempotency key and verify it returns the same logical reservation without double-counting capacity.
5. Attempt to reserve more than the remaining quantity and verify the API rejects it.
6. Cancel the reservation through `/api/store-drops/:dropId/cancel-reservation` and verify remaining quantity is released from current reservation rows.
7. Pause and close the merchant drop, then verify new reservations are blocked.
8. Run `/api/jobs/expire-store-drops` and verify old pickup windows move to terminal status with `job_runs` and `audit_events` evidence.

Expected result: all final states come from live rows generated by these actions, not from seed output.

## Browser Smoke

Open these pages on the production alias:

- `/drops`: live surplus drops, remaining quantity, pickup windows, reserve/cancel state, sold-out/expired states, and no-payment wording.
- `/merchant`: surplus create/edit/publish/pause/close controls, active reservations, remaining capacity, pickup windows, and heatmap summary.
- `/proof`: CP7 row counts, audit/job evidence, demo controls, no-payment status, and heatmap privacy proof.

Expected result: UI reads live routes or shows unavailable states. It must not expose exact household coordinates, unit labels, direct personal contact fields, or raw household-level need locations.

## Aurora Smoke

Verify directly or through `/api/system/state`:

- Published drops are counted from `store_drops.status = 'published'`.
- Active reservations are counted from current non-terminal `store_drop_reservations` rows.
- Sold-out, closed, and expired drops are counted as terminal.
- Heatmap cells are coarse aggregate rows or route output, not household locations.
- `expire-store-drops` creates `job_runs` rows.
- CP7 mutations create `audit_events` rows.

Expected result: no response or UI surface claims Stripe, deposits, card capture, payment authorization, held funds, ledger entries, or paid commitments. Any price is display/demo intent only.
