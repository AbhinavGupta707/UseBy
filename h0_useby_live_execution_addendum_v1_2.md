# H0 UseBy + DemandPool — Live Execution Addendum v1.2

**Purpose:** Convert the existing UseBy PRD from a strong product/architecture spec into an explicit live-working implementation plan for H0.  
**Core rule:** Seeded data is allowed for the demo world, but outcomes must be computed live from Aurora PostgreSQL state. Do not ship pre-baked action cards, fixed match results, fixed booking transitions, or static merchant awards.

---

## 1. Current assessment

The existing UseBy PRD v1.1 already has many of the right execution mechanics:

- Aurora PostgreSQL as primary database.
- PostGIS for neighbourhood matching.
- State machines for grocery items, rental items, DemandPool, and merchant bids.
- Next.js/Vercel Server Actions and API route handlers.
- Vercel Cron Jobs for expiry decay, match recomputation, pool closure, and reminders.
- Seeded neutral neighbourhood with fictional merchants.
- Transactional booking and reservation logic to prevent double booking.
- Action cards that reflect backend state.
- Merchant portal and DemandPool as a secondary module.

However, it is not yet as explicit as the RunwayOps live-agentic plan about what is truly live versus seeded. This addendum makes that explicit.

---

## 2. Live-product standard

For H0, UseBy should demonstrate a working local-resource engine, not a scripted clickable prototype.

### Allowed synthetic/demo data

Seed these at database setup:

- one neutral fictional neighbourhood
- 8 households
- 2–3 merchants
- sample grocery items
- sample wardrobe/household items
- sample needs/intents
- sample merchant catalogue/drop data
- one parsed receipt fixture
- one parsed expiry-label fixture
- one sample GS1 Digital Link string

### Not allowed as cached outputs

Do not seed these as fixed final outputs:

- final action cards
- final match ranking
- booking state transitions after user action
- merchant bid winner
- trust score changes after completion
- expiry confidence updates after new observations
- demand-pool threshold transitions

Those must be generated by server-side code from current database rows.

---

## 3. Product execution loop

The live loop should be:

```text
seeded / uploaded input
  -> Aurora row created
  -> deterministic rules + PostGIS + optional AI wording
  -> action card / match / pool state written to Aurora
  -> user acts through Server Action
  -> transactional state transition
  -> audit event written
  -> recompute affected matches/cards/pools
  -> UI refreshes from live database state
```

This matters because the judges should be able to click a button, change the state, and see backend-derived consequences rather than pre-recorded UI changes.

## 4. Core live mechanics by module

## 4.1 Receipt and inventory ingestion

### MVP live path

1. User clicks **Upload receipt** or **Use demo receipt**.
2. If demo receipt is used, load the parsed receipt fixture as input, not as final cards.
3. Server route writes `receipt_imports`, `receipt_line_items`, and `item_instances` rows.
4. Expiry estimator creates `expiry_observations` rows with confidence bands.
5. `recompute_action_cards(household_id)` runs and writes new `action_cards` rows.
6. UI fetches current cards from Aurora.

### Proof of liveness

- Upload/import the same receipt twice should be idempotent or should create distinct imports depending on selected mode.
- Changing storage state from cupboard to fridge/freezer should update expiry band and action cards.
- Manually entering/scanning an expiry date should replace estimated expiry with confirmed expiry.

### Required tables

- `receipt_imports`
- `receipt_line_items`
- `item_instances`
- `expiry_observations`
- `inventory_events`
- `action_cards`

## 4.2 Action-card generation

Action cards must be rule-driven first, optionally AI-polished second.

### Deterministic card engine

Generate cards from database state:

- `use_first`: item is near estimated or confirmed expiry.
- `scan_label`: item has low expiry confidence.
- `share_sealed`: sealed item is near expiry and a nearby need exists.
- `rent_out`: wardrobe/household item has availability and nearby demand.
- `borrow_instead`: user has a need and local supply exists.
- `join_pool`: a demand pool exists for a relevant recurring purchase.
- `merchant_drop`: a local merchant drop matches household preferences.

### Optional AI use

Use an LLM only to convert a deterministic card into consumer-friendly language:

```json
{
  "card_type": "share_sealed",
  "facts": {
    "item": "sealed tortilla wraps",
    "expiry_band": "use soon",
    "matched_need_distance_m": 420,
    "trust_score": 0.86
  }
}
```

The model returns copy only, not the decision.

### Proof of liveness

- Add a new need nearby and the `share_sealed` card appears.
- Mark the item consumed and the card disappears.
- Change expiry label and the card priority changes.

## 4.3 Matching engine

### Inputs

- item category
- expiry urgency
- availability window
- need category
- distance
- trust score
- pickup compatibility
- food safety restrictions
- item state

### Backend query pattern

Use PostGIS for radius queries:

```sql
SELECT
  i.id AS item_id,
  n.id AS need_id,
  ST_Distance(i.location_geom, n.location_geom) AS distance_m,
  i.expiry_confidence,
  i.state,
  n.needed_by
FROM item_instances i
JOIN needs n
  ON i.category = n.category
WHERE i.state IN ('private', 'use_soon', 'offered')
  AND n.status = 'open'
  AND ST_DWithin(i.location_geom, n.location_geom, n.radius_m)
  AND i.safety_status = 'eligible'
ORDER BY distance_m ASC;
```

Then calculate a match score and write `matches` rows.

### Proof of liveness

- Create a need and see matches generated.
- Move household radius smaller/larger and see matches change.
- Reserve one item and see it removed from competing matches.

## 4.4 Booking and handoff state machine

This is the most important anti-cached mechanic.

### State machine

```text
requested -> accepted -> reserved -> pickup_scheduled -> picked_up -> completed -> reviewed
                    \-> declined
                    \-> cancelled
```

### Transaction requirement

Booking acceptance must be transactional:

```text
BEGIN;
  SELECT item_instances.id, state
  FROM item_instances
  WHERE id = :item_id
  FOR UPDATE;

  if state not available -> rollback and return conflict

  INSERT booking
  UPDATE item_instances SET state = 'reserved'
  INSERT inventory_event
  INSERT audit_event
COMMIT;
```

### Proof of liveness

- Two users try to reserve the same dress or item.
- The first succeeds; the second gets a live conflict or alternative suggestion.
- Item availability window updates in the UI.

## 4.6 DemandPool mechanics

DemandPool should be secondary in the demo, but still live.

### State machine

```text
draft -> gathering -> threshold_met -> bidding -> awarded -> ready_for_pickup -> fulfilled
      \-> expired
```

### Live flow

1. User joins a pool and commits quantity/price ceiling.
2. Aurora updates `pool_commitments`.
3. If threshold is met, server changes pool state to `bidding`.
4. Merchant submits a bid.
5. Award algorithm evaluates price, distance, pickup window, capacity, reliability, and substitutions.
6. Winning bid is written to `merchant_bids` and `demand_pools.awarded_bid_id`.
7. Pickup tasks are generated.

### Award scoring

```text
score =
  price_score * 0.35 +
  pickup_window_score * 0.20 +
  distance_score * 0.15 +
  fulfilment_reliability * 0.20 +
  substitution_quality * 0.10
```

### Proof of liveness

- Add one more household commitment and watch the pool cross threshold.
- Submit two merchant bids and see the award recomputed.
- Change a merchant pickup window and see the winning bid change.

## 4.7 Cron jobs and background execution

Use Vercel Cron Jobs for scheduled work.

### Jobs

| Job | Route | Purpose |
|---|---|---|
| Expiry decay | `/api/jobs/expiry-decay` | Recompute expiry bands and urgent cards daily |
| Match recompute | `/api/jobs/recompute-matches` | Refresh matches and action cards |
| Pool closure | `/api/jobs/close-demand-pools` | Move pools to bidding/expired/awarded states |
| Pickup reminders | `/api/jobs/pickup-reminders` | Create/send reminder notifications |

### Idempotency

Each job writes to:

- `job_runs`
- `audit_events`

Use idempotency keys:

```text
job_type + neighbourhood_id + window_start
```

### Proof of liveness

Add a “System activity” panel showing:

- last expiry job run
- cards updated
- matches recomputed
- pools transitioned
- latest audit events

## 5. Live demo mode design

Use a guided demo, but not a pre-canned UI.

### Demo control buttons

- **Reset neighbourhood**: truncates demo rows and reseeds initial state.
- **Import demo receipt**: writes receipt lines and item instances.
- **Add neighbour need**: inserts a live `needs` row.
- **Run matching**: calls recompute endpoint.
- **Accept booking**: executes transaction.
- **Join DemandPool**: inserts commitment and checks threshold.
- **Submit merchant bid**: writes live bid.
- **Award pool**: runs award algorithm.

Each button should call a real server route and write to Aurora.

### Judge-visible proof panel

Add a small panel called **Live System State**:

- Aurora database connected
- latest mutation timestamp
- item count
- open needs
- active matches
- active bookings
- active demand pools
- latest audit event
- last cron/job run

This panel is extremely useful because it visibly differentiates the app from a static v0 mockup.

## 7. Aurora implementation requirements

### Extensions

```sql
CREATE EXTENSION IF NOT EXISTS postgis;
CREATE EXTENSION IF NOT EXISTS vector;
```

`pgvector` is optional for MVP but useful for matching free-text needs to item descriptions.

### Required transaction boundaries

Use DB transactions for:

- accepting a booking
- completing a booking
- joining a demand pool
- awarding a merchant bid
- changing item state from available to reserved
- writing action outcome + trust event

### Audit trail

Every meaningful action writes an `audit_events` row:

- who acted
- what changed
- before/after state
- source route/job
- timestamp
- idempotency key where relevant

## 8. What not to overbuild

For H0, do not over-prioritise:

- real payment settlement
- full merchant onboarding
- real grocery API integrations
- real identity verification
- full food-safety compliance
- complex recommender models
- native mobile app

The winning version is not “complete marketplace.” It is a beautiful, working proof that private household inventory can become live local economic action.

## 10. Final H0 positioning

UseBy should be described as:

> A live neighbourhood resource engine. Seeded households and merchants provide the demo world, but every action card, match, booking, pool threshold, merchant award, and trust event is computed from Aurora PostgreSQL state through Vercel server actions, route handlers, cron jobs, and PostGIS queries.

The strongest judge-facing claim is:

> This is not a marketplace mockup. It is a small but working local economy with real backend state transitions.
